# RLang v0.2 — Complete Language Specification

## A Rust-Inspired Agentic Reasoning Language

**Specification v0.2 — April 2026**
**Status: Design Approved, Pre-Implementation**

---

## 1. Executive Summary

RLang is a strongly-typed, Rust-inspired reasoning language designed as the internal
scratchpad for AI agents. It replaces natural language chain-of-thought with structured,
typed, verifiable reasoning traces.

v0.2 expands the language from a belief-reasoning scratchpad (v0.1) into a complete
agentic reasoning language covering four domains: epistemic state, motivational state,
operational execution, and inter-agent communication.

### 1.1 Design Drivers (Research-Backed)

This specification is grounded in empirical analysis of:

- **5.68M+ reasoning traces** from NVIDIA OpenMathReasoning, DeepSeek R1, and Claude
- **7 structural reasoning patterns** identified across ReAct, Reflexion, ToT, GoT, Inner Monologue
- **6 productive micro-patterns** from OpenAI o1 analysis (Wu et al. 2024)
- **Token efficiency data**: only 40-60% of English CoT tokens are productive; 27-51% are
  removable self-reflection markers
- **A2A Protocol v1.0** (Google/Linux Foundation) for inter-agent communication
- **FIPA-ACL** (22 speech acts), **BDI logic**, **Contract Net Protocol**, and
  **Agent Contracts** (Ye & Tan 2025) for multi-agent formal foundations

### 1.2 Core Goals

1. **3-5x token compression** over English reasoning (empirically achievable per research)
2. **Structural error prevention** — 7 known failure modes caught by the type system
3. **Complete agentic coverage** — beliefs, goals, actions, and inter-agent communication
4. **Rust-proximate syntax** — dual-use for agents that both reason and generate Rust code
5. **A2A protocol alignment** — native expression of agent discovery, delegation, and task lifecycle
6. **Formal grammar** — PEG-parseable, validator-checkable

## 2. Language Architecture

### 2.1 Four-Layer Type System

RLang organizes all reasoning into four layers. Each layer has its own types,
operators, and validation rules. Layers compose — higher layers reference lower ones.

```
┌─────────────────────────────────────────────────────────────────┐
│  Layer 4: COMMUNICATIVE — Inter-agent messages and coordination │
│  How do I coordinate with other agents?                        │
├─────────────────────────────────────────────────────────────────┤
│  Layer 3: OPERATIONAL — Actions, tools, feedback               │
│  What do I do in the world?                                    │
├─────────────────────────────────────────────────────────────────┤
│  Layer 2: MOTIVATIONAL — Goals, intentions, plans              │
│  What do I want to achieve?                                    │
├─────────────────────────────────────────────────────────────────┤
│  Layer 1: EPISTEMIC — Beliefs, evidence, confidence            │
│  What do I know and how certain am I?                          │
└─────────────────────────────────────────────────────────────────┘
```

### 2.2 Mandatory Reasoning Phases

Every reasoning trace must pass through four phases in order. The parser enforces
phase structure. Skipping a phase or unbounded looping is a structural error.

```rust
#[phase]
enum Phase {
    Frame,    // Reformulate problem, extract constraints, identify context
    Explore,  // Decompose, generate candidates, apply methods
    Verify,   // Check work, bounded backtrack, test against constraints
    Decide,   // Assert conclusion, select action, render output
}
```

**Phase transition rules:**
- `Frame -> Explore`: mandatory, exactly once
- `Explore -> Verify`: mandatory, exactly once
- `Verify -> Explore`: allowed, bounded by `max_rebloom: u8` (default: 3)
- `Verify -> Decide`: mandatory terminal transition
- `Decide -> _`: no further transitions (terminal phase)

### 2.3 Rust Alignment Principles

Every RLang construct mirrors a Rust concept. This is not cosmetic — it ensures
models trained on RLang internalize Rust's correctness guarantees.

| Rust Concept | RLang Usage |
|---|---|
| `enum` (sum types) | All operator results, task states, communicative acts |
| `struct` | All compound types (beliefs, goals, contracts) |
| `Result<T, E>` | All fallible operations return `Ok<T>` or `Err<E>` |
| `Option<T>` | All optional fields use `Some<T>` or `None` |
| `trait` / `impl` | Reasoning modes declared as trait implementations |
| `match` | All branching is exhaustive pattern matching |
| Lifetimes (`'a`) | Information freshness: `'fresh`, `'stale`, `'unk` |
| Ownership | Beliefs are owned by the reasoning trace; delegation transfers ownership |
| `#[derive]`, `#[cfg]` | Attributes for phase markers, validation hints |
| `mod` | Namespacing for domain-specific operator extensions |
| `where` clauses | Constraints on generic types (e.g., `where T: Verifiable`) |

## 3. Layer 1: Epistemic — What Do I Know?

The epistemic layer models beliefs, evidence, and confidence. This is the foundation
all other layers build on. Expanded from v0.1 with research-driven additions.

### 3.1 Core Types

```rust
/// A typed belief with confidence, epistemic mode, and freshness lifetime.
/// The fundamental epistemic unit. Every claim must be wrapped in a Blf.
struct Blf<'a, T> {
    claim: T,                    // The proposition being believed
    p: f64,                      // Confidence: 0.0..=1.0 (required, never implicit)
    ep: EpMode,                  // How the belief was formed
    src: Src,                    // Where it came from
    scope: Scope,                // How broadly it applies
    fresh: Lifetime,             // 'fresh | 'stale | 'unk — Rust lifetime analog
}

/// Epistemic mode — how was this belief formed?
enum EpMode {
    Direct,   // Directly observed (ep:direct)
    Infer,    // Logically inferred from other beliefs (ep:infer)
    Anl,      // Analogical reasoning from similar cases (ep:anl)
    Recv,     // Received from another agent (ep:recv) — NEW in v0.2
}

/// Source of a belief
enum Src {
    Obs(ObsId),          // From direct observation
    Chain(Vec<BlfId>),   // Inferred from chain of beliefs
    Agent(AgentId),      // Received from another agent — NEW in v0.2
    Mem(MemId),          // Retrieved from memory — NEW in v0.2
    Given,               // Provided as input/premise
}

/// Scope of a claim — how broadly does it apply?
enum Scope {
    All,    // Universal: "all X are Y"
    Some,   // Existential: "some X are Y"
    None,   // Negative universal: "no X are Y"
    Cond,   // Conditional: "if P then Q"
    Gen,    // General tendency
    Loc,    // Local to this context only
}

/// Information freshness — Rust lifetime analog
enum Lifetime {
    Fresh,   // 'fresh — recently observed/verified
    Stale,   // 'stale — not recently verified, may have changed
    Unk,     // 'unk — freshness unknown
}

/// Evidence supporting or weakening a belief
struct Evidence {
    items: Vec<EvidenceItem>,
}

/// A single piece of evidence with its effect on a belief
enum EvidenceItem {
    Sup(BlfId, f64),   // sup(belief, +delta) — supports, increases confidence
    Wkn(BlfId, f64),   // wkn(belief, -delta) — weakens, decreases confidence
    Neut(BlfId),       // neut(belief) — relevant but neither supports nor weakens
}
```

### 3.2 Core Operators (13 total)

All operators return typed results. No operator produces untyped output.

```rust
// === Causal Operators ===

/// a causes b
fn cause<A, B>(a: A, b: B) -> Blf<Causal<A, B>>;

/// a prevents b
fn prvnt<A, B>(a: A, b: B) -> Blf<Causal<A, B>>;

/// a enables b (makes b possible, does not guarantee it)
fn enbl<A, B>(a: A, b: B) -> Blf<Causal<A, B>>;

/// a requires b (a cannot exist/succeed without b)
fn req<A, B>(a: A, b: B) -> Blf<Dependency<A, B>>;

// === Evidence Operators ===

/// x is directly observed
fn obs<T>(x: T) -> Blf<Observation<T>>;

/// a is similar to b (analogical link)
fn sim<A, B>(a: A, b: B) -> Blf<Analogy<A, B>>;

/// a conflicts with b (contradiction detected)
fn confl<A, B>(a: A, b: B) -> Blf<Conflict<A, B>>;

// === State Operators ===

/// x changes from state a to state b
fn chng<X, A, B>(x: X, from: A, to: B) -> Blf<StateChange<X, A, B>>;

/// x is negated or canceled
fn cncl<T>(x: T) -> Blf<Negation<T>>;

// === Structure Operators ===

/// a contains b
fn cntns<A, B>(a: A, b: B) -> Blf<Containment<A, B>>;

/// a is a type/category of b
fn isa<A, B>(a: A, b: B) -> Blf<Classification<A, B>>;

// === Temporal Operators ===

/// a occurs before/then b (temporal sequence)
fn seq<A, B>(a: A, b: B) -> Blf<Sequence<A, B>>;

// === Intent Operators ===

/// agent's goal is to achieve x
fn goal<A, X>(agent: A, target: X) -> Blf<Goal<A, X>>;
```

### 3.3 Belief Resolution

Beliefs are resolved through evidence evaluation, returning Rust-style `Result`.

```rust
// Declare a belief
let claim: Blf<0.7> = cause(storm, cncl(mtg))
    | ep:infer | src:obs(weather_rpt) | scope:loc | t:fresh;

// Build evidence block
let ev: Evidence = [
    obs(dark_clouds)    => sup(claim, +0.20),
    obs(no_notice)      => wkn(claim, -0.15),
    obs(empty_lot)      => sup(claim, +0.10),
];

// Resolve — returns Result
claim |> resolve(ev) -> match {
    Ok(resolved) => resolved,  // Blf with updated confidence
    Err(e) => match e {
        InsufEv     => suspend(claim),    // Not enough evidence
        Conflict(a, b) => confl(a, b),    // Contradictory evidence
        Stale(blf)  => blf.decay(0.15),   // Stale evidence degrades
    },
}
```

### 3.4 Confidence Branching

All branching is exhaustive `match` — no silent fallthrough.

```rust
match conf(claim) {
    c if c > 0.85 => assert(claim),              // High confidence: commit
    c if c > 0.55 => hedge(claim, cond: [..]),   // Medium: conditional assertion
    c if c > 0.30 => suspend(claim),             // Low: hold, gather more evidence
    _             => reject(claim),              // Very low: discard
}
```

### 3.5 Lifetime Decay and Refresh

Stale beliefs cannot be asserted without explicit handling.

```rust
let old_data: Blf<0.9, 'stale> = cause(bug, crash);

// Option 1: Decay confidence (acknowledge staleness)
let degraded: Blf<0.75, 'stale> = old_data.decay(0.15);

// Option 2: Refresh with new observation (restore freshness)
let refreshed: Blf<0.9, 'fresh> = old_data.refresh(obs(new_log));

// Compile error: cannot assert stale belief directly
// assert(old_data);  // ERROR: Blf<_, 'stale> does not impl Assertable
```

### 3.6 Trait-Based Reasoning Modes

Reasoning mode is declared explicitly before reasoning begins.

```rust
impl Deductive for ReasoningTrace {
    // Conclusion follows necessarily from premises
    fn reason(&self, premises: &[Blf]) -> Result<Blf, DeductiveErr> {
        if enbl(key, access) && obs(key_present) {
            Ok(assert(access) | p:0.99 | ep:direct)
        } else {
            Err(DeductiveErr::PremiseMissing)
        }
    }
}

impl Abductive for ReasoningTrace {
    // Best explanation for observed evidence
    fn reason(&self, observations: &[Obs]) -> Result<Blf, AbductiveErr> {
        obs(crash) => Ok(cause?(mem_leak) | p:0.72 | ep:infer)
    }
}

impl Analogical for ReasoningTrace {
    // Transfer from known case to new case
    fn reason(&self, source: &Case, target: &Case) -> Result<Blf, AnalogicalErr> {
        sim(proj_a, proj_b) | p:0.6
            => Ok(transfer(fix_a -> fix_b) | ep:anl)
    }
}
```

## 4. Layer 2: Motivational — What Do I Want?

The motivational layer models goals, desires, intentions, and plans. Draws from
BDI (Belief-Desire-Intention) logic formalized by Rao & Georgeff (1995).

### 4.1 Core Types

```rust
/// What the agent wants to achieve. Not yet committed to.
struct Desire<T> {
    target: T,                 // What we want
    priority: Priority,        // How important
    rationale: Vec<BlfId>,     // Why we want it (grounded in beliefs)
}

/// A desire that has been evaluated and selected for pursuit.
/// Has a defined target, success criteria, and resource bounds.
struct Goal<T> {
    target: T,                 // What to achieve
    priority: Priority,        // Urgency/importance ranking
    deadline: Deadline,        // Temporal constraint
    success: Predicate<T>,     // How to know we achieved it
    status: GoalStatus,        // Current state
    parent: Option<GoalId>,    // Parent goal if this is a subgoal
}

enum Priority {
    Critical,   // Must be achieved, blocks everything else
    High,       // Should be achieved soon
    Normal,     // Standard priority
    Low,        // Nice to have, do if resources permit
    Background, // Only if nothing else needs doing
}

enum Deadline {
    Urgent,                     // ASAP — minimal reasoning budget
    By(Timestamp),              // Hard deadline
    Within(Duration),           // Relative deadline
    Flexible,                   // No time pressure
    None,                       // No deadline
}

enum GoalStatus {
    Pending,     // Not yet started
    Active,      // Currently being pursued
    Achieved,    // Success criteria met
    Failed(Reason),   // Cannot be achieved
    Abandoned(Reason), // Voluntarily given up
    Blocked(Vec<GoalId>), // Waiting on subgoals/dependencies
}

/// A goal the agent has committed to, with an attached plan.
/// Intent = Goal + Plan + Resource Commitment.
struct Intent<T> {
    goal: Goal<T>,
    plan: Plan,
    resources: ResourceBudget,   // What we're willing to spend
    progress: usize,             // Current step index in plan
}

/// An ordered sequence of steps to achieve a goal.
struct Plan {
    steps: Vec<Step>,
    dependencies: DAG<StepId>,  // Which steps depend on which
    status: PlanStatus,
}

enum Step {
    Reason(ReasoningTask),       // Think about something (epistemic)
    Act(Action),                 // Do something in the world (operational)
    Delegate(DelegationSpec),    // Hand off to another agent (communicative)
    Verify(VerificationSpec),    // Check a result
    Branch(MatchExpr),           // Conditional branching
    SubPlan(Plan),               // Nested plan (recursive decomposition)
}

enum PlanStatus {
    Draft,        // Not yet validated
    Ready,        // Validated, ready to execute
    InProgress,   // Currently executing
    Completed,    // All steps done
    Failed(StepId, Reason),  // Failed at specific step
    Replanned,    // Abandoned in favor of new plan
}
```

### 4.3 Resource Budgets

Every intent carries a resource budget. Delegation must satisfy conservation laws.

```rust
struct ResourceBudget {
    tokens: u64,             // Max reasoning tokens
    api_calls: u32,          // Max external API calls
    wall_time: Duration,     // Max wall-clock time
    delegations: u8,         // Max sub-delegations allowed
    cost: Option<Currency>,  // Max monetary cost
}

// Conservation law — enforced by type system
// When delegating, child budgets must sum to <= parent budget
#[invariant]
fn resource_conservation(parent: &Intent, children: &[Intent]) -> bool {
    children.iter().map(|c| c.resources).sum() <= parent.resources
}
```

## 5. Layer 3: Operational — What Do I Do?

The operational layer models actions, tool use, environmental feedback, memory,
and self-correction. Draws from ReAct, Reflexion, and Inner Monologue patterns.

### 5.1 Core Types

```rust
/// An action the agent can take in the world.
/// Actions are not beliefs — they change state rather than describing it.
enum Action<T> {
    /// Invoke an external tool
    Invoke {
        tool: ToolId,
        args: T,
        expected: Blf<f64>,      // Expected outcome (belief about what will happen)
    },
    /// Execute a primitive operation
    Exec {
        op: Operation,
        args: T,
    },
    /// Delegate to another agent (bridges to Layer 4)
    Delegate {
        to: AgentId,
        task: Task,
        contract: Contract,
    },
    /// Store to memory
    Remember {
        key: MemKey,
        value: T,
        mem_type: MemType,
    },
    /// Retrieve from memory
    Recall {
        query: Query,
        mem_type: MemType,
    },
    /// Emit output to user/consumer
    Emit {
        content: T,
        format: OutputFormat,
    },
}

/// Result of executing an action — always typed
enum ActionResult<T> {
    Ok(T),                         // Success with typed output
    Err(ActionErr),                // Failure with reason
    Partial(T, Vec<Remaining>),    // Partial success
    Timeout,                       // Exceeded time budget
    Blocked(BlockReason),          // Cannot proceed, needs input
}

/// Types of memory — from A-MEM research (2025)
enum MemType {
    Episodic,     // Specific past events and experiences
    Semantic,     // General knowledge and facts
    Procedural,   // How to do things (skills, plans)
    Working,      // Current context / scratchpad
}

/// Environmental feedback received after an action
struct ObsFeed<T> {
    data: T,                       // What was observed
    source: FeedSource,            // Where it came from
    timestamp: Timestamp,          // When it was received
    relevance: f64,                // How relevant to current goal (0.0..=1.0)
}

enum FeedSource {
    Env,        // Direct environmental observation
    Tool(ToolId),  // Response from a tool invocation
    Agent(AgentId), // Feedback from another agent
    User,       // Human-in-the-loop input
}
```

### 5.2 The Observe-Think-Act Loop

The fundamental agentic control structure. RLang expresses it as a bounded loop
with mandatory observation before each action.

```rust
/// The core agent loop. Always bounded — no infinite loops.
#[bounded(max_iterations)]
fn agent_loop<G>(
    goal: Goal<G>,
    max_iterations: u8,
) -> Result<G, AgentErr> {
    let mut state = State::init();

    for _i in 0..max_iterations {
        // OBSERVE: mandatory perception before reasoning
        let obs: ObsFeed<_> = perceive(state.env);

        // THINK: reason about observation + goal + memory
        let plan: Plan = reason(obs, goal, state.memory)?;

        // ACT: execute next step
        let result: ActionResult<_> = execute(plan.next_step())?;

        // UPDATE: integrate feedback
        state = state.update(result);

        // CHECK: are we done?
        if goal.success.check(&state) {
            return Ok(state.extract_result());
        }
    }

    Err(AgentErr::MaxIterations(max_iterations))
}
```

### 5.3 Self-Correction (Bounded Backtrack)

From Reflexion research. Backtracking is structural, not prose. Always bounded.

```rust
/// A structured reflection on a failed attempt.
/// NOT free-form text — typed diagnosis with specific fields.
struct Reflection {
    attempt: TraceId,                 // Which attempt failed
    outcome: ActionResult<()>,        // What happened
    diagnosis: DiagnosisKind,         // Why it failed (typed, not prose)
    revision: Option<Plan>,           // New plan if available
}

enum DiagnosisKind {
    WrongApproach(Reason),            // Method was unsuitable
    MissingInfo(Vec<Query>),          // Need more data
    ConstraintViolation(Constraint),  // Broke a requirement
    ToolFailure(ToolId, ToolErr),     // External tool failed
    InsufficientEvidence,             // Not enough to conclude
    ConflictingEvidence(ConflId),     // Evidence contradicts itself
}

/// Bounded backtrack — max retries enforced by type system
#[bounded(max_retries: u8 = 3)]
fn backtrack<T>(
    failed: Reflection,
    max_retries: u8,
) -> Result<T, BacktrackErr> {
    // Each retry MUST have a novel strategy
    // Type system enforces: revision.plan != previous.plan
    match failed.diagnosis {
        WrongApproach(reason) => {
            let new_plan = replan(failed.attempt, reason)?;
            // new_plan must differ from failed plan — checked by validator
            execute(new_plan)
        },
        MissingInfo(queries) => {
            let data = queries.iter().map(|q| recall(q, Semantic)).collect();
            retry_with(failed.attempt, data)
        },
        _ => Err(BacktrackErr::Unrecoverable(failed.diagnosis)),
    }
}
```

### 6.2 Communicative Acts

Drawn from FIPA-ACL's 22 speech acts, compressed to the 12 most common in
agentic communication. Each is a typed variant, not free text.

```rust
/// A typed communicative act between agents.
/// Every inter-agent message is one of these — no untyped messages.
enum CommAct {
    // === Informative ===
    /// Assert a proposition to another agent
    Inform { to: AgentId, content: Blf<()> },
    /// Confirm a previously communicated proposition
    Confirm { to: AgentId, claim: BlfId },
    /// Deny a previously communicated proposition
    Disconfirm { to: AgentId, claim: BlfId, reason: Reason },

    // === Requestive ===
    /// Request another agent to perform an action
    Request { to: AgentId, action: Action<()>, deadline: Deadline },
    /// Ask whether a proposition is true
    QueryIf { to: AgentId, proposition: Blf<()> },

    // === Negotiation ===
    /// Call for proposals — broadcast task to candidate agents
    Cfp { to: Vec<AgentId>, task: TaskSpec, conditions: Constraints },
    /// Propose to perform a task under conditions
    Propose { to: AgentId, action: Action<()>, conditions: Constraints },
    /// Accept a proposal
    Accept { to: AgentId, proposal: ProposalId },
    /// Reject a proposal with reason
    Reject { to: AgentId, proposal: ProposalId, reason: Reason },

    // === Commitment ===
    /// Commit to performing an action
    Agree { to: AgentId, action: Action<()>, contract: Contract },
    /// Decline to perform an action
    Refuse { to: AgentId, action: Action<()>, reason: Reason },

    // === Lifecycle ===
    /// Report task failure
    Failure { to: AgentId, task: TaskId, reason: Reason },
}
```

### 6.3 Task Lifecycle — A2A State Machine

Maps directly to the A2A Protocol's 8 task states with enforced valid transitions.

```rust
/// Task state — discriminated union matching A2A Protocol v1.0
enum TaskState {
    Submitted,
    Working { progress: Option<f64> },
    InputRequired { prompt: Query },
    AuthRequired { scheme: SecurityScheme },
    Completed { artifacts: Vec<Artifact> },
    Failed { error: TaskErr },
    Canceled { reason: Reason },
    Rejected { reason: Reason },
}

/// Valid state transitions — enforced by the type system.
/// Attempting an invalid transition is a compile-time error.
impl TaskState {
    fn valid_transitions(&self) -> Vec<TaskState> {
        match self {
            Submitted => vec![Working, Rejected, Canceled],
            Working   => vec![Completed, Failed, Canceled, InputRequired, AuthRequired],
            InputRequired => vec![Working, Canceled],
            AuthRequired  => vec![Working, Canceled],
            // Terminal states — no further transitions
            Completed | Failed | Canceled | Rejected => vec![],
        }
    }
}

/// A2A-aligned task with context threading
struct Task {
    id: TaskId,
    context_id: ContextId,       // Groups related interactions
    status: TaskState,
    artifacts: Vec<Artifact>,
    history: Vec<Message>,
    metadata: HashMap<String, Value>,
}

/// Multi-part message — mirrors A2A Message type
struct Message {
    id: MessageId,
    context_id: Option<ContextId>,
    task_id: Option<TaskId>,
    role: Role,
    parts: Vec<Part>,
    reference_tasks: Vec<TaskId>,  // Cross-references to related tasks
}

enum Role {
    User,    // Message from client/delegator
    Agent,   // Message from server/delegate
}

/// Message content — tagged union matching A2A Part type
enum Part {
    Text(String),
    Data(Value),                     // Structured JSON data
    Raw { bytes: Vec<u8>, media_type: String, filename: Option<String> },
    Url { url: String, media_type: Option<String> },
}

/// Task output container
struct Artifact {
    id: ArtifactId,
    name: Option<String>,
    description: Option<String>,
    parts: Vec<Part>,
}
```

### 6.4 Agent Contracts

### 6.5 Trust Model

From the FIRE trust model (Huynh et al. 2006). Four trust sources.

```rust
/// Multi-source trust model
struct TrustModel {
    /// Direct experience with this agent
    interaction: HashMap<AgentId, f64>,
    /// Trust based on agent's role in the system
    role_based: HashMap<RoleId, f64>,
    /// Third-party reports about agent reliability
    witness: HashMap<AgentId, Vec<(AgentId, f64)>>,
    /// Formally attested credentials
    certified: HashMap<AgentId, Vec<Certificate>>,
}

impl TrustModel {
    /// Compute aggregate trust score for an agent
    fn trust_score(&self, agent: &AgentId) -> f64 {
        let direct   = self.interaction.get(agent).unwrap_or(&0.5);
        let role     = self.role_based.get(&agent.role).unwrap_or(&0.5);
        let witness  = self.witness_average(agent);
        let cert     = self.cert_score(agent);
        weighted_combine(*direct, *role, witness, cert)
    }

    /// Bayesian update after interacting with an agent
    fn update(&mut self, agent: &AgentId, outcome: &TaskOutcome) {
        let current = self.interaction.entry(*agent).or_insert(0.5);
        *current = bayesian_update(*current, outcome.quality());
    }
}

/// Trust-gated delegation — cannot delegate to untrusted agents
fn delegate_trusted<T>(
    task: Task,
    candidate: AgentId,
    trust: &TrustModel,
    min_trust: f64,
) -> Result<TaskId, DelegationErr> {
    let score = trust.trust_score(&candidate);
    if score < min_trust {
        return Err(DelegationErr::InsufficientTrust {
            agent: candidate,
            score,
            required: min_trust,
        });
    }
    // Proceed with delegation...
    Ok(delegate(task, candidate))
}
```

## 7. Connective Operators

Connectives wire expressions together. Two from v0.1, six new in v0.2.

```rust
// === Sequential ===
|>    // Pipe: pass result of left to right (from v0.1)
->    // Transform/resolve: left produces right (from v0.1)

// === Parallel ===
||>   // Fan-out: execute multiple branches in parallel
<|    // Aggregate: merge multiple results into one

// === Exploratory ===
~>    // Tentative: exploratory step (can be reverted without cost)

// === Error ===
!>    // Error channel: route errors to handler
?>    // Fallible: try left, on failure try right (like Rust's ?)

// === Memory ===
@>    // Store: pipe result into memory
<@    // Retrieve: pull from memory into pipeline
```

### 7.1 Connective Examples

```rust
// Sequential pipeline
obs(logs) |> analyze() |> cause?(bug, crash) -> resolve(ev)

// Parallel fan-out + aggregation
decompose(goal) ||> [
    delegate(agent_a, subtask_1),
    delegate(agent_b, subtask_2),
    delegate(agent_c, subtask_3),
] <| synthesize(results)

// Tentative exploration (revertible)
hypothesis ~> test(hypothesis) -> match {
    Ok(confirmed) => assert(confirmed),
    Err(_) => backtrack(hypothesis),
}

// Error channel
execute(deploy) !> match {
    ToolFailure(e) => replan(deploy, e),
    Timeout => retry_with(deploy, extended_deadline),
    _ => escalate(deploy),
}

// Memory operations in pipeline
obs(user_preference) @> remember("prefs", Semantic)
recall("prefs", Semantic) <@ |> apply_to(response)
```

## 8. Anti-Pattern Prevention

The grammar and type system structurally prevent the 7 known reasoning failure modes.

### 8.1 Failure Mode Prevention Table

| # | Failure Mode | Prevention Mechanism | Enforcement Level |
|---|---|---|---|
| 1 | **Circular reasoning** | Phase system: cannot re-enter Frame after Explore | Parser (grammar) |
| 2 | **Infinite reflection** | `#[bounded(max_retries)]` on all backtrack operations | Type system |
| 3 | **Constraint forgetting** | `req()` constraints persist across all phases, checked at Decide | Validator |
| 4 | **Premature commitment** | Verify phase is mandatory before Decide | Parser (grammar) |
| 5 | **Over-verification** | Single Verify phase; rebloom count bounded by `max_rebloom` | Parser + config |
| 6 | **Rumination** | Backtrack requires `DiagnosisKind` + novel `revision.plan` | Type system |
| 7 | **Confidence theater** | `p:` computed from evidence via `resolve()`, not asserted | Type system |

### 8.2 Structural Rules

```rust
// Rule 1: Confidence must be computed, not asserted
// INVALID: let claim: Blf<0.9> = cause(a, b);  // Where does 0.9 come from?
// VALID:   let claim = cause(a, b) |> resolve(evidence) -> Ok(Blf<0.9>);

// Rule 2: Backtrack must have novel strategy
#[invariant]
fn novel_backtrack(previous: &Plan, revision: &Plan) -> bool {
    revision != previous  // Structural inequality, not just parameter changes
}

// Rule 3: Phase ordering enforced
#[invariant]
fn valid_phase_sequence(trace: &[Phase]) -> bool {
    // Must follow: Frame -> Explore -> Verify -> Decide
    // With optional bounded Verify -> Explore rebounds
    trace.windows(2).all(|w| valid_transition(w[0], w[1]))
}

// Rule 4: Resource conservation in delegation
#[invariant]
fn delegation_conservation(parent: &Contract, children: &[Contract]) -> bool {
    children.iter().map(|c| &c.resources).sum::<ResourceBudget>() <= parent.resources
}

// Rule 5: Terminal task states are final
#[invariant]
fn terminal_is_final(state: &TaskState) -> bool {
    match state {
        Completed { .. } | Failed { .. } | Canceled { .. } | Rejected { .. } => {
            // No further state transitions allowed
            true
        },
        _ => false,
    }
}
```

## 10. Complete Reasoning Trace Example

A full multi-layer reasoning trace for: *"Should we deploy the fix, and can we
delegate the monitoring to Agent-B?"*

```rust
#[phase(Frame)]
impl Deductive {

    // === LAYER 1: EPISTEMIC — What do I know? ===
    let tests: blf<0.99> = obs(tests_pass)
        | ep:direct | src:ci_pipeline | scope:loc | t:fresh;
    let risk: blf<0.85> = obs(no_rollback_plan)
        | ep:direct | src:obs(infra_check) | scope:loc | t:fresh;
    let traffic: blf<0.90> = obs(low_traffic)
        | ep:direct | src:obs(metrics) | scope:loc | t:fresh;

    // === LAYER 2: MOTIVATIONAL — What do I want? ===
    let deploy_goal: goal<Deploy> = goal(self, deploy(fix))
        | priority:high | deadline:within(2h)
        | success: obs(health_check_pass);

    let monitor_goal: goal<Monitor> = goal(self, monitor(deploy))
        | priority:normal | deadline:within(24h);
}

#[phase(Explore)]
{
    // Resolve deploy confidence
    let ev: Evidence = [
        tests    => sup(deploy_goal, +0.15),
        risk     => wkn(deploy_goal, -0.25),
        traffic  => sup(deploy_goal, +0.10),
    ];

    let deploy_blf = enbl(fix, resolve(bug))
        |> resolve(ev) -> Ok(blf<0.70>);

    // === LAYER 4: COMMUNICATIVE — Can Agent-B monitor? ===
    let agent_b: AgentCard = discover("monitoring")
        |> match_capability(monitor_goal);

    let trust_b: f64 = trust_model.trust_score(&agent_b.id);
    // trust_b = 0.82 — sufficient for monitoring (threshold: 0.6)

    let monitor_contract = Contract {
        input_spec: schema!(deploy_id, health_endpoint),
        output_spec: schema!(alert_on_failure),
        resources: ResourceBudget { wall_time: 24h, api_calls: 100 },
        success: obs(no_alerts_24h) | obs(alert_handled),
        mode: Balanced,
    };
}

#[phase(Verify)]
{
    // Check constraints
    req(deploy_goal, obs(tests_pass)) |> verify(tests) -> Ok(());
    req(monitor_goal, trust_b >= 0.6) |> verify(trust_b) -> Ok(());

    // Verify resource conservation
    let total_budget = self.intent.resources;
    let monitor_budget = monitor_contract.resources;
    assert(monitor_budget <= total_budget);  // Conservation law holds
}

#[phase(Decide)]
{
    // Branch on deploy confidence
    match conf(deploy_blf) {
        c if c > 0.80 => assert(deploy_goal),
        c if c > 0.55 => hedge(deploy_goal, cond: [
            enbl(rollback_plan, deploy_goal),
        ]),
        _ => reject(deploy_goal),
    }
    // -> hedge: deploy only if rollback plan exists

    // Delegate monitoring to Agent-B
    dlg(agent_b.id, monitor_goal, monitor_contract)
        -> Ok(task_id: "mon-001");

    // Emit decision
    emit(Decision {
        deploy: Hedge { condition: "rollback plan required" },
        monitoring: Delegated { to: agent_b.id, task: "mon-001" },
    });
}
```

**Rendered English output:** "The fix is ready and tests pass, but I recommend
deploying only once a rollback plan is in place. I've delegated 24-hour post-deploy
monitoring to Agent-B (trust: 0.82), who will alert on any health check failures."

## 11. Grammar Summary (PEG Target)

The Pest PEG grammar will parse the following top-level structure:

```
trace       = { phase_block+ }
phase_block = { phase_attr ~ impl_block? ~ statement+ }
phase_attr  = { "#[phase(" ~ phase_name ~ ")]" }
phase_name  = { "Frame" | "Explore" | "Verify" | "Decide" }

statement   = { let_binding | match_expr | pipe_chain | fn_call | assertion }
let_binding = { "let" ~ pattern ~ (":" ~ type_ann)? ~ "=" ~ expr ~ metadata? ~ ";" }
match_expr  = { "match" ~ expr ~ "{" ~ match_arm+ ~ "}" }
pipe_chain  = { expr ~ (connective ~ expr)+ }
fn_call     = { ident ~ "(" ~ args ~ ")" }
assertion   = { ("assert" | "hedge" | "suspend" | "reject") ~ "(" ~ expr ~ ")" }

metadata    = { "|" ~ meta_field ~ ("|" ~ meta_field)* }
meta_field  = { meta_key ~ ":" ~ meta_value }
meta_key    = { "p" | "ep" | "src" | "scope" | "t" | "priority" | "deadline" }
meta_value  = { float | ident | fn_call }

connective  = { "|>" | "->" | "||>" | "<|" | "~>" | "!>" | "?>" | "@>" | "<@" }

expr        = { fn_call | literal | ident | match_expr | block | pipe_chain }
```

## 12. Open Design Questions

1. **Module system**: Should domain-specific operator libraries (medical, legal, financial)
   use Rust-style `mod` + `use` imports?
2. **Async semantics**: Should streaming/push notifications use `async/await` syntax
   or callback-style handlers?
3. **Generics depth**: How deep should generic type parameters go? `Blf<Causal<Obs<T>, Goal<U>>>`
   could get unwieldy.
4. **Macro system**: Should RLang support `macro_rules!`-style macros for common patterns?
5. **Interop format**: What serialization format for traces? JSON? Protobuf? Custom binary?
6. **Validation strictness levels**: Should there be `#[allow]`/`#[warn]`/`#[deny]`
   like Rust's lint system?
7. **Rendering phase spec**: How exactly does the RLang -> English rendering work?
   Separate model? Template system? Both?
